# Defense Criteria Guide

## 1. Problem Definition & Objectives

### What This Criterion Means

This checks if the project has a clear problem, a valid purpose, and objectives that match the actual system developed.

### What The IT Panelists Will Look For

- Is the problem clear and real?
- Is it relevant to the IT field?
- Do the objectives match the system?
- Did the group build what the study promised?

### Strong Simple Line

“Our project addresses the limitation of traditional LMS platforms by developing an adaptive learning management system that identifies student learning preferences and transforms learning materials into more personalized content.”

## 2. System Implementation & Functionality

### What This Criterion Means

This checks if the system actually works, if the important features are complete, if the parts are connected properly, and if the system is usable and reliable.

### What The IT Panelists Will Look For

- Are the core features complete?
- Do the modules work together properly?
- Is the system reliable during actual use?
- Is it easy enough for users to use?

### Strong Simple Line

“Our system implements the core functions of a learning management system while integrating adaptive features that work together to deliver personalized learning content in a usable and reliable platform.”

## 3. Innovation / Technical Contribution

### What This Criterion Means

This checks what makes the project technically different, useful, and not just a basic LMS.

### What The IT Panelists Will Look For

- Is the solution new or improved?
- Are the technologies used appropriate?
- Does the system give a real technical contribution to the problem?

### Strong Simple Line

“Our project goes beyond a traditional LMS by combining learning style identification, machine learning, and AI-based content transformation to create a more adaptive and personalized learning experience.”

## 4. Documentation Quality

### What This Criterion Means

This checks if the paper and technical documentation are organized, complete, clear, and consistent.

### What The IT Panelists Will Look For

- Is the documentation well-organized?
- Are the important parts complete?
- Is the explanation clear and easy to follow?
- Is the content consistent with the actual system?

### Strong Simple Line

“Our documentation clearly presents the problem, objectives, methodology, system design, implementation, and results in an organized and complete manner.”

### Important Methodology Note

One current risk is that the methodology may describe what was used, but not clearly how it was actually done from start to finish.
